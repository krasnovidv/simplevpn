# Auto-reconnect race condition kills VPN tunnel on connect

**Date:** 2026-03-09 15:10
**Files:** mobile/app/android/app/src/main/kotlin/com/simplevpn/app/SimpleVpnService.kt, pkg/transport/ws/ws.go
**Severity:** critical

## Problem

VPN connects successfully (TLS + WebSocket + credential auth all OK), status shows "connected" for ~6 seconds, then connection dies with "Recv error (after 0 packets): read frame length: read frame header: software caused connection abort". Zero packets are exchanged.

## Root Cause

Race condition in `SimpleVpnService.kt` when auto-reconnect is enabled:

1. `connect()` calls `builder.establish()` creating TUN1 (fd=42)
2. Android fires `onLost` for the physical network (VPN establishment changes routing)
3. `onLost` blindly sets `currentStatus = "disconnected"` — even though Go tunnel is alive
4. Go side finishes auth, sets status to "connected"
5. Android fires `onAvailable` — sees `currentStatus = "disconnected"` (stale)
6. Auto-reconnect triggers after backoff delay
7. `connect()` calls `builder.establish()` AGAIN — this **invalidates the old TUN fd**
8. `Vpnlib.connect()` returns "already connected" error
9. `finally` block closes the new TUN
10. Go library is reading from the now-invalid old fd — tunnel dies
11. Server gets no data, eventually closes connection — client sees TCP RST

Secondary bug: `bufio.NewReader` data loss during WebSocket upgrade. The temporary `bufio.Reader` used for HTTP parsing may buffer data beyond the HTTP headers. When `WrapClient/WrapServer(conn)` wraps the raw connection, any buffered data is lost.

## Solution

1. **Guard `connect()` against re-entry**: Check Go-side status before calling `builder.establish()`. If Go reports "connected" or "connecting", skip the reconnect.

2. **Fix `onLost` callback**: Check Go-side status before setting `currentStatus = "disconnected"`. VPN establishment itself triggers `onLost` for the physical network — this is normal, not a real disconnection.

3. **Fix `onAvailable` callback**: Use Go-side status (`Vpnlib.status()`) instead of stale `currentStatus` to decide whether to auto-reconnect.

4. **Fix WebSocket upgrade**: Transfer any data buffered by `bufio.Reader` to the WS conn's `readBuf` after HTTP parsing, preventing data loss.

## Prevention

- Always use the Go-side status as the source of truth for VPN connection state
- Android network callbacks fire during VPN establishment — never assume `onLost` means the VPN is down
- When using `bufio.Reader` temporarily, always check `Buffered()` and transfer remaining data
- Test auto-reconnect on real devices (network callbacks behave differently on emulators)
