# WebSocket upgrade rejected — server returns decoy page instead of 101

**Date:** 2026-03-08 23:50
**Files:** Dockerfile, cmd/server-hardened/main.go, deploy/deploy.sh
**Severity:** critical

## Problem

Mobile client sends WebSocket upgrade request to the VPN server. Server responds with HTTP 200 and decoy HTML page (`<html><body><h1>Welcome</h1><p>Service running.</p></body></html>`) instead of 101 Switching Protocols. VPN connection fails immediately.

## Root Cause

**Deployment gap**: The server binary on 193.23.3.93 (deployed 2026-03-06) predates the WebSocket transport commits (`e326035`, `f681dd5`). The old binary has no WS detection — it reads the `GET /ws HTTP/1.1...` request as a 56-byte auth token, HMAC verification fails, and it falls through to `serveDecoy()`.

Two blockers prevented redeployment:
1. **Dockerfile used Go 1.21** but `go.mod` requires Go 1.25 (added with uTLS dependency) — Docker build would fail
2. **Deploy script didn't rebuild** — `docker compose up -d` reuses cached images, so code changes never reach the container

Additionally, `serveWebSocket()` bypassed `PeekConn.Read()` by manually passing `conn.Conn` + peeked data to `ServerUpgrade`, which was fragile.

## Solution

1. Updated Dockerfile Go version: `golang:1.21-alpine` → `golang:1.25-alpine`
2. Added `--build` flag to deploy script: `docker compose up -d --build vpn-server`
3. Simplified `serveWebSocket()` to pass PeekConn directly to `ServerUpgrade` — PeekConn.Read() transparently replays peeked bytes
4. Added hex logging of peeked bytes in transport detection for future debugging

## Prevention

- When updating `go.mod` version (e.g., adding dependencies that require newer Go), always update Dockerfile Go version to match
- Deploy scripts should always force rebuild (`--build`) to avoid stale images
- After adding new transport/protocol features, redeploy the server before testing from mobile
- Use the peeked bytes hex log to quickly diagnose transport detection issues
