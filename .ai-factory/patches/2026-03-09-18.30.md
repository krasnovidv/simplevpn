# Phone reboot from TUN fd use-after-free and uncontrolled reconnection

**Date:** 2026-03-09 18:30
**Files:** mobile/vpnlib/vpnlib.go, mobile/app/android/app/src/main/kotlin/com/simplevpn/app/SimpleVpnService.kt, mobile/app/lib/services/vpn_service.dart
**Severity:** critical

## Problem

Phone reboots when user connects VPN, switches to another app (e.g. YouTube), then returns to the VPN app. The crash is a native segfault/SIGABRT caused by Go reading/writing a closed TUN file descriptor.

## Root Cause

Multiple cascading issues:

1. **TUN fd use-after-free:** Android network callbacks (`onAvailable`/`onLost`) fire when app switches, triggering reconnection. A second `builder.establish()` invalidates the old TUN fd while Go goroutines are still reading/writing it. The Go `tunFile.Read()` or `tunFile.Write()` hits a closed fd, causing a native crash that propagates through gomobile to JVM and triggers a system reboot.

2. **No fd cleanup:** `Disconnect()` and `cleanup()` never closed `tunFile`, leaving fd leaked. Goroutines blocked in `tunFile.Read()` couldn't be unblocked on disconnect.

3. **No panic recovery:** TUN reader goroutine had no `recover()`, so any panic killed the entire Go runtime and crashed the app.

4. **No connect synchronization:** Multiple concurrent `Connect()` calls could race, and multiple reconnect threads spawned from network callbacks without any coordination.

5. **No debounce on network callbacks:** Rapid WiFi/cellular changes spawned unlimited reconnect threads via `Thread { Thread.sleep(); connect() }.start()`.

6. **Aggressive Flutter polling:** 2-second status polling continued when app was backgrounded, wasting resources and potentially interfering with service lifecycle.

## Solution

1. **Go vpnlib:** Added `connectMu` mutex to prevent concurrent `Connect()` calls. Added `tunFile.Close()` in both `Disconnect()` and `cleanup()` to release fd and unblock stuck goroutines. Added `defer recover()` in TUN reader goroutine to catch panics gracefully.

2. **Kotlin service:** Added `AtomicBoolean isConnecting` to prevent concurrent `connect()` calls. Replaced raw Thread-based reconnection with `Handler.postDelayed()` with 500ms debounce. Cancel pending reconnects in `removeNetworkMonitoring()`.

3. **Flutter service:** Added `WidgetsBindingObserver` to pause polling when app is backgrounded and resume on foreground. Increased poll interval from 2s to 5s.

## Prevention

- Always close file descriptors in cleanup/disconnect paths — never assume the platform will do it
- Goroutines that use native resources (fd, syscalls) MUST have `defer recover()` to prevent Go runtime crashes
- Network callbacks on Android fire during VPN establishment — always debounce and use Go-side status as source of truth
- Use `AtomicBoolean` or mutex for connect/disconnect operations, never allow concurrent execution
- Pause background polling when app is not visible
