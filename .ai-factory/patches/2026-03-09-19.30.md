# Double HTTP response on failed WebSocket upgrade + missing diagnostic logging

**Date:** 2026-03-09 19:30
**Files:** pkg/transport/ws/ws.go
**Severity:** medium

## Problem

When a non-WebSocket GET request hit the server (scanner/bot or potentially a VPN client with issues), the server sent TWO HTTP responses: first `HTTP/1.1 400 Bad Request` from `ServerUpgrade()`, then `HTTP/1.1 200 OK` from `serveDecoy()`. This is invalid HTTP and makes the server fingerprintable by DPI systems (real nginx never sends double responses). Additionally, no diagnostic info was logged about what headers were actually received, making it impossible to distinguish scanner traffic from real VPN client failures.

## Root Cause

`ServerUpgrade()` in `ws.go` was writing a 400 response directly to the connection before returning an error. The caller (`serveWebSocket` in `main.go`) then called `serveDecoy()` which wrote a 200 response to the same connection. The function violated separation of concerns — it should only validate and return errors, not write responses.

## Solution

1. Removed the `conn.Write([]byte("HTTP/1.1 400 Bad Request\r\n\r\n"))` calls from `ServerUpgrade()`. The caller handles the decoy response.
2. Added diagnostic logging that shows the actual HTTP method, path, Upgrade header, and Host when a non-WebSocket request is rejected. This helps distinguish scanner traffic from VPN client issues.

## Prevention

- Functions that validate/parse should not write responses directly. Keep response writing in the caller.
- Always log enough context on error paths to support debugging without reproducing the issue.
- When implementing decoy/stealth behavior, test that only ONE valid HTTP response is sent per connection.
